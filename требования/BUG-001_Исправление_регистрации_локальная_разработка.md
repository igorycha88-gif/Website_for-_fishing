# Bug Report: Исправление работы регистрации при локальной разработке

## Обзор

**ID**: BUG-001
**Version**: 1.0
**Author**: System Analyst
**Date**: 2026-02-12
**Priority**: High (критично для разработки)

## Проблема

**Описание**: Функционал регистрации перестал работать при локальной разработке (frontend запущен через `npm run dev`).

**Текущее поведение**:
- Пользователь заполняет форму регистрации
- Frontend отправляет POST запрос на `http://localhost:3000/api/v1/auth/register`
- Получает ошибку `400 (Bad Request)`
- Регистрация не выполняется

**Ожидаемое поведение**:
- Пользователь заполняет форму регистрации
- Frontend отправляет POST запрос на `http://localhost:3000/api/v1/auth/register`
- Next.js проксирует запрос на auth-service (port 8001)
- Пользователь успешно регистрируется

## Логи ошибки

```
POST http://localhost:3000/api/v1/auth/register 400 (Bad Request)
```

## Root Cause Analysis

**Корневая причина**: Конфигурация Next.js использует `host.docker.internal` для проксирования API запросов, но при локальной разработке (frontend запущен через `npm run dev`, не в Docker) этот хост не резолвится.

**Технические детали**:

| Компонент | Хост | Порт |
|-----------|------|------|
| Frontend (Next.js) | localhost | 3000 |
| Auth Service (FastAPI) | localhost | 8001 |

**Текущая конфигурация** (next.config.js):
```javascript
{
  source: '/api/v1/auth/:path*',
  destination: 'http://host.docker.internal:8001/api/v1/auth/:path*',
}
```

**Проблема**: `host.docker.internal` предназначен для доступа к хосту изнутри Docker контейнера, но не работает при локальной разработке (frontend запущен напрямую через Node.js).

## Предлагаемое решение

### Вариант 1: Условная конфигурация на основе переменной окружения (РЕКОМЕНДУЕТСЯ)

**Описание**: Использовать переменную окружения для определения режима работы (Docker или локальная разработка).

**Реализация**:

1. Создать переменную окружения `DOCKER_ENV` (по умолчанию `false`)

2. Обновить `next.config.js`:
```javascript
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: "standalone",
  images: {
    domains: ['localhost', 'host.docker.internal'],
  },
  async rewrites() {
    const isDocker = process.env.DOCKER_ENV === 'true';
    const authHost = isDocker ? 'http://host.docker.internal:8001' : 'http://localhost:8001';
    const placesHost = isDocker ? 'http://host.docker.internal:8002' : 'http://localhost:8002';
    const reportsHost = isDocker ? 'http://host.docker.internal:8003' : 'http://localhost:8003';
    const bookingHost = isDocker ? 'http://host.docker.internal:8004' : 'http://localhost:8004';
    const shopHost = isDocker ? 'http://host.docker.internal:8005' : 'http://localhost:8005';
    const emailHost = isDocker ? 'http://host.docker.internal:8006' : 'http://localhost:8006';

    return [
      {
        source: '/api/v1/auth/:path*',
        destination: `${authHost}/api/v1/auth/:path*`,
      },
      {
        source: '/api/v1/users/:path*',
        destination: `${authHost}/api/v1/users/:path*`,
      },
      {
        source: '/api/v1/maps/:path*',
        destination: `${authHost}/api/v1/maps/:path*`,
      },
      {
        source: '/api/v1/places/:path*',
        destination: `${placesHost}/api/v1/places/:path*`,
      },
      {
        source: '/api/v1/reports/:path*',
        destination: `${reportsHost}/api/v1/reports/:path*`,
      },
      {
        source: '/api/v1/booking/:path*',
        destination: `${bookingHost}/api/v1/booking/:path*`,
      },
      {
        source: '/api/v1/shop/:path*',
        destination: `${shopHost}/api/v1/shop/:path*`,
      },
      {
        source: '/api/v1/email/:path*',
        destination: `${emailHost}/api/v1/email/:path*`,
      },
    ];
  },
};

module.exports = nextConfig;
```

3. Обновить `docker-compose.yml` для установки переменной:
```yaml
frontend:
  environment:
    - DOCKER_ENV=true
```

4. Обновить `.env.example` для локальной разработки:
```env
DOCKER_ENV=false
```

**Плюсы**:
- Простая реализация
- Работает в обоих режимах (Docker и локальная разработка)
- Минимальные изменения в коде
- Сохраняет текущую архитектуру

**Минусы**:
- Требуется обновить docker-compose.yml

### Вариант 2: Использование NEXT_PUBLIC_API_URL с полным URL

**Описание**: Использовать полную URL для API endpoints вместо проксирования через Next.js.

**Реализация**:

1. Обновить `frontend/app/lib/api.ts`:
```typescript
const API_URL = process.env.NEXT_PUBLIC_API_URL || "http://localhost:8001";

export const API_ENDPOINTS = {
  AUTH: {
    LOGIN: `${API_URL}/api/v1/auth/login`,
    REGISTER: `${API_URL}/api/v1/auth/register`,
    VERIFY_EMAIL: `${API_URL}/api/v1/auth/verify-email`,
    RESET_PASSWORD: `${API_URL}/api/v1/auth/reset-password/request`,
  },
  // ... остальные endpoints
};
```

2. Обновить `.env` для Docker:
```env
NEXT_PUBLIC_API_URL=http://host.docker.internal:8001
```

3. Удалить или закомментировать rewrites в `next.config.js`

**Плюсы**:
- Прямые запросы к сервисам
- Нет проксирования через Next.js

**Минусы**:
- Требует изменений в архитектуре
- Может нарушить CORS настройки
- Менее гибкое решение

---

### Вариант 3: Использование nginx для проксирования (критически сложный)

**Описание**: Добавить nginx как reverse proxy для проксирования запросов к микросервисам.

**Реализация**:
- Добавить nginx в docker-compose.yml
- Настроить nginx для проксирования запросов по пути `/api/*`
- Frontend обращается к nginx на порту 8080

**Плюсы**:
- Единая точка входа
- Стандартный паттерн для микросервисов

**Минусы**:
- Критически сложный для текущей стадии
- Требует значительных изменений в архитектуре
- Увеличивает инфраструктуру

## Требования к реализации

### Функциональные требования

**FR1**: Регистрация должна работать при локальной разработке
- **Given**: Frontend запущен через `npm run dev`
- **When**: Пользователь заполняет форму регистрации с валидными данными
- **Then**: Запрос успешно проксируется на auth-service
- **And**: Пользователь создается в базе данных
- **And**: Возвращается успешный ответ с сообщением о регистрации

**FR2**: Регистрация должна работать в Docker окружении
- **Given**: Frontend запущен в Docker контейнере
- **When**: Пользователь заполняет форму регистрации с валидными данными
- **Then**: Запрос успешно проксируется на auth-service
- **And**: Пользователь создается в базе данных
- **And**: Возвращается успешный ответ с сообщением о регистрации

**FR3**: Все API endpoints должны работать корректно
- **Given**: Frontend запущен в любом режиме (Docker или локальный)
- **When**: Выполняется запрос к любому API endpoint
- **Then**: Запрос успешно проксируется на соответствующий сервис
- **And**: Возвращается корректный ответ

### Не-функциональные требования

**Performance**: Проксирование не должно значительно увеличивать время отклика (target: < 50ms overhead)

**Compatibility**: Решение должно работать как на macOS (host.docker.internal), так и на Windows и Linux

**Maintainability**: Код должен быть легко читаемым и поддерживаемым

**Testing**: Решение должно быть протестировано в обоих режимах (Docker и локальный)
