# ЧТЗ — INC-SEC-001: Ликвидация компрометации frontend-контейнера (криптомайнер/бэкдор)

- **Тип**: Инцидент безопасности + инфраструктура
- **Маршрут**: Аналитик → DevOps → Тестировщик (постдеплой)
- **Серьёзность**: Критическая (P0)
- **Среда**: Production VPS `5.35.102.219`, `/opt/fishmap`, compose `docker-compose.vps-host.yml`
- **Дата обнаружения**: 2026-06-29

## 1. Описание инцидента

Сайт `http://5.35.102.219/` недоступен. Диагностика выявила компрометацию контейнера `fishmap-frontend-1`.

## 2. Причины (root cause)

1. **`network_mode: host`** у сервиса `frontend` → Next.js слушает `0.0.0.0:3000` на хосте.
2. **Firewall отсутствует** (`iptables -P INPUT ACCEPT`, правил нет, UFW не установлен) → порт 3000 **доступен из интернета напрямую**, минуя nginx.
3. Через прямой доступ к Next.js эксплуатирована RCE в рантайме → в `/tmp` контейнера дропнут бэкдор `astro.sh4` (54 КБ) + `dropbear` (SSH-бэкдор), запущены сотни маскированных процессов (`[cpuhp/2]`, `[ssl_client]`).
4. Синхронные `execSync`-вызовы вредоноса заблокировали Node.js event loop → Next.js перестал отвечать (HTTP 000) → nginx upstream timeout → **сайт лежит**.

## 3. Масштаб компрометации (на момент расследования)

- ❌ Контейнер `fishmap-frontend-1`: заражён (`/tmp/astro.sh4`, `dropbear.lock`, 363 маскированных процесса, попытки скачать XMRig — провалились с Permission denied).
- ✅ Исходный код и собранный образ **ЧИСТЫЕ** (нет `execSync`/`child_process`/`xmrig` в `.next` и git).
- ✅ Хост: без cron-персистенции, без новых пользователей, SSH-ключ только `github-actions-deploy` (требует верификации).
- ✅ Майнер фактически не майнит (загрузка скачиваний падает).

## 4. Критерии приёмки

1. Сайт `http://5.35.102.219/` отдаёт HTTP 200.
2. Порт 3000 **недоступен извне** (внешний curl → отказ/timeout, не accept).
3. Контейнер frontend пересоздан из чистого образа; в `/tmp` нет `astro.sh4`/`dropbear`.
4. Количество процессов в контейнере в норме (нет `[cpuhp/2]`/`[ssl_client]`).
5. Firewall настроен: извне открыты только 22 (SSH), 80, 443; остальное DROP. SSH-доступ сохранён.
6. Пароль `root` ротирован. SSH-ключ `github-actions-deploy` верифицирован (удалён если чужой).
7. Логи сервисов без ошибок; healthcheck frontend → healthy.
8. Улики сохранены в `/opt/fishmap/forensics/`.

## 5. Декомпозиция задач

- **TASK-SEC-01** [DevOps]: Настроить firewall (безопасно, с аварийным откатом) — закрыть 3000 извне.
- **TASK-SEC-02** [DevOps]: Пересоздать `fishmap-frontend-1` из чистого образа.
- **TASK-SEC-03** [DevOps]: Ротация пароля root; аудит SSH-ключей.
- **TASK-SEC-04** [Аналитик/Security]: Криминалистика `astro.sh4`; определение точного вектора (Next.js CVE / SSR / зависимость).
- **TASK-SEC-05** [DevOps]: Харденинг compose (убрать `network_mode: host` или сохранить firewall персистентно; read-only `/tmp`).
- **TASK-SEC-06** [DevOps]: Полная пересборка сервисов (Правило 6).
- **TASK-SEC-07** [Тестировщик]: Постдеплойная проверка на живом стенде (API/E2E/логи).

## 6. Исполнитель

DevOps (основные операции) + Security review (криминалистика) + Тестировщик (постдеплой).

## 7. Безопасность операции

- Перед любыми деструктивными действиями улики скопированы в `/opt/fishmap/forensics/YYYYMMDD/`.
- Firewall применяется с аварийным откатом (фоновый flush через timeout), чтобы не потерять SSH.
- Правила iptables персистируются только после проверки доступности.
