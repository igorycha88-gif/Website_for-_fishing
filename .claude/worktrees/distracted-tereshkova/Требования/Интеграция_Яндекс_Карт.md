# Требования к интеграции с Яндекс Картами для сайта FishMap

## Введение и контекст

### Краткое описание проекта

**Название проекта:** FishMap — платформа для рыболовов

**Тип проекта:** Микросервисное веб-приложение для управления местами для рыбалки

**Технический стек:**
- **Frontend:** Next.js 15 (App Router), React 18, TypeScript, Tailwind CSS, Zustand (state management)
- **Backend:** FastAPI (Python), микросервисная архитектура
- **База данных:** PostgreSQL 14+
- **Инфраструктура:** Docker Swarm, Traefik (API Gateway)
- **Карта:** Яндекс Карты JavaScript API (API Key: dfb59053-0011-47fb-a6f1-a14efb9160d1)

**Архитектурные компоненты:**
- Auth Service (порт 8001) — аутентификация и пользователи
- Places Service (порт 8002) — управление местами для рыбалки
- Reports Service (порт 8003) — отчеты рыбаков
- Booking Service (порт 8004) — бронирование
- Shop Service (порт 8005) — магазин
- Email Service (порт 8006) — почтовые уведомления

**Текущее состояние:**
- Таблица `places` в базе данных создана и содержит необходимые поля
- Frontend компоненты `MyPlacesTab` и страница `/map` созданы, но не имеют функционала
- Places Service имеет базовую структуру, но без бизнес-логики

- Возраст: 35 лет

### Сценарии использования (Use Cases)

#### UC1: Сохранение личного места
**Актор:** Авторизованный пользователь

**Основной поток:**
1. Пользователь авторизуется в системе
2. Открывает вкладку «Мои места»
3. Нажимает кнопку «Добавить место»
4. На Яндекс Карте кликает на нужную точку
5. Заполняет форму:
   - Название места
   - Описание (обязательно)
   - Типы рыб (выбор из фиксированного списка)
   - Фото (до 5 штук)
   - Дата последнего посещения
   - Дополнительные удобства
6. Выбирает тип доступа: «Личное» или «Публичное»
7. Нажимает «Сохранить»
8. Система сохраняет место и показывает подтверждение

**Альтернативные потоки:**
- 4a. Пользователь вводит координаты вручную
- 6a. Если выбрано «Публичное», место отправляется на модерацию

---

#### UC2: Просмотр публичных мест на карте
**Актор:** Авторизованный пользователь

**Основной поток:**
1. Пользователь открывает страницу «Карта»
2. Яндекс Карта загружается с центром на его геолокации
3. На карте отображаются все прошедшие модерацию публичные места
4. Пользователь применяет фильтры (тип рыбы, рейтинг, расстояние)
5. Карта обновляется с учетом фильтров
6. Пользователь кликает на метку места
7. Открывается баллон с детальной информацией

**Альтернативные потоки:**
- 2a. Если геолокация недоступна, центр карты — Москва
- 3a. При большом количестве мест используются кластеры

---

#### UC3: Модерация публичного места
**Актор:** Администратор

**Основной поток:**
1. Администратор открывает панель модерации
2. Видит список мест, ожидающих модерации
3. Просматривает детали места (карта, фото, описание)
4. Принимает решение: «Одобрить» или «Отклонить»
5. Если одобрено — место становится публичным
6. Если отклонено — владелец получает уведомление

## Функциональные требования

### Модуль 1: Управление местами (Places Service)

#### FR1.1: Создание места для рыбалки

**User Story:**
> Как авторизованный пользователь, я хочу добавить новое место для рыбалки на карту, чтобы сохранить его для будущих рыбалок и, при желании, поделиться с другими рыболовами.

**Критерии приемки (Acceptance Criteria):**

**Дано (Given):**
- Пользователь авторизован
- Открыта вкладка «Мои места» или страница «Карта»

**Когда (When):**
- Пользователь нажимает кнопку «Добавить место»
- Выбирает точку на Яндекс Карте (кликом или вводом координат)
- Заполняет обязательные поля:
  - Название места (2-200 символов)
  - Описание (10-5000 символов)
  - Типы рыб (минимум 1 вид из фиксированного списка)
- Выбирает тип доступа: «Личное» или «Публичное»
- Загружает от 0 до 5 фотографий (JPEG, PNG, до 10MB каждая)
- Опционально заполняет:
  - Дата последнего посещения
  - Удобства (парковка, туалет, баня, WiFi и др.)
  - Дополнительные комментарии
- Нажимает кнопку «Сохранить»

**Тогда (Then):**
- Место сохраняется в базе данных со статусом:
  - «Личное» — сразу активно
  - «Публичное» — статус «На модерации»
- Если место публичное — отправляется уведомление администраторам
- Отображается сообщение об успехе
- Место появляется на карте пользователя
- Если тип доступа «Публичное» — владелец видит статус «На модерации»

**Валидация:**
- Координаты должны быть валидны (широта: -90 до 90, долгота: -180 до 180)
- Название не может быть пустым
- Описание обязательно
- Типы рыб — минимум 1 вид
- Фото — до 5 штук, валидные форматы
- Если публичное место — необходимо минимум 1 фото

**API Endpoint:**
```
POST /api/v1/places
Headers: Authorization: Bearer <token>
Body: {
  "title": "string (2-200 chars)",
  "description": "string (10-5000 chars)",
  "latitude": "decimal (10,8)",
  "longitude": "decimal (11,8)",
  "fish_types": ["fish1", "fish2", ...], // из фиксированного списка
  "images": ["url1", "url2", ...], // до 5
  "visit_date": "ISO 8601 date",
  "facilities": ["parking", "toilet", ...], // JSONB
  "access_type": "private" | "public"
}
Response: 201 Created
{
  "id": "uuid",
  "status": "active" | "pending_moderation",
  ...
}
```

### Модуль 2: Яндекс Карты интеграция (Frontend)

#### FR2.1: Инициализация Яндекс Карты

**User Story:**
> Как пользователь, я хочу видеть интерактивную карту с моими и публичными местами для рыбалки, чтобы визуально оценить их расположение.

**Критерии приемки (Acceptance Criteria):**

**Дано (Given):**
- Пользователь авторизован
- Открыта страница «Карта»

**Когда (When):**
- Страница загружается
- Яндекс Карты JS API загружается асинхронно

**Тогда (Then):**
- Карта инициализируется с ключом API: `dfb59053-0011-47fb-a6f1-a14efb9160d1`
- Центр карты определяется автоматически:
  - Если доступна геолокация — центр на пользователе
  - Если нет — Москва (55.7558, 37.6173)
- Масштаб карты: 10 (зум)
- Отображаются:
  - Все активные публичные места
  - Личные места текущего пользователя
- Метки отображаются с цветовой кодировкой:
  - Синие — личные места
  - Красные — публичные места
  - Зеленые — выбранные/активные места

**Технические требования:**
- Использовать Яндекс Карты JS API v2.1 (или актуальную версию)
- Подключение через CDN: `https://api-maps.yandex.ru/2.1/?lang=ru_RU&apikey=YOUR_API_KEY`
- Инициализация после загрузки скрипта
- Обработка ошибок загрузки API

**Пример кода:**
```javascript
ymaps.ready(init);

### Модуль 3: Модерация мест

#### FR3.1: Отправка на модерацию

**User Story:**
> Как пользователь, я хочу чтобы мои публичные места проходили проверку перед публикацией, чтобы убедиться в качестве контента на платформе.

**Критерии приемки (Acceptance Criteria):**

**Дано (Given):**
- Пользователь авторизован
- Пользователь создает или редактирует место
- Выбран тип доступа «Публичное»

**Когда (When):**
- Пользователь сохраняет место
- Место проходит валидацию

**Тогда (Then):**
- Место сохраняется со статусом `status = 'pending_moderation'`
- Отправляется уведомление всем администраторам
- Владелец видит статус «На модерации»
- Место НЕ отображается на публичной карте
- Владелец может просматривать место на своей карте

---

#### FR3.2: Панель модерации для администратора

**User Story:**
> Как администратор, я хочу видеть все места, ожидающие модерации, в одном месте с возможностью быстрого одобрения или отклонения.

**Критерии приемки (Acceptance Criteria):**

**Дано (Given):**
- Администратор авторизован
- У администратора есть роль `admin` или `moderator`

**Когда (When):**
- Администратор открывает панель модерации

**Тогда (Then):**
- Отображается список всех мест со статусом `pending_moderation`
- Каждое место показывает:
  - Превью фото
  - Название и описание
  - Координаты и карту (мини)
  - Типы рыб
  - Информацию о владельце
  - Дату создания
- Доступны быстрые действия:
  - Одобрить (значок галочки)
  - Отклонить (значок креста)
- При одобрении:
  - Статус меняется на `active`
  - Отправляется уведомление владельцу
  - Место появляется на публичной карте
- При отклонении:
  - Статус меняется на `rejected`
  - Запрашивается причина отклонения (обязательно)
  - Отправляется уведомление владельцу с причиной

**API для модерации:**
```
POST /api/v1/places/:id/moderate
Headers: Authorization: Bearer <admin_token>
Body: {
  "action": "approve" | "reject",
  "reason": "string" // обязательно при reject
}
Response: 200 OK
```

---

#### FR3.3: Критерии авто-модерации

**User Story:**
> Как система, я хочу автоматически отклонять места, которые явно не соответствуют правилам, чтобы сократить нагрузку на модераторов.

**Критерии приемки (Acceptance Criteria):**

**Авто-отклонение происходит если:**
- Название содержит запрещенные слова (мат, спам)
- Описание содержит запрещенные слова
- Координаты находятся в недопустимых зонах (океан, границы)
- Типы рыб не выбраны или выбраны недопустимые
- Отсутствует фото (для публичных мест)

**При авто-отклонении:**
- Статус: `rejected`
- Причина: автоматически сгенерирована
- Уведомление отправляется владельцу

**API:**
```
// При создании места проверяется авто-модерация
POST /api/v1/places
// Если не проходит авто-модерацию:
Response: 400 Bad Request
{
  "error": {
    "code": "AUTO_REJECTED",
    "message": "Место не прошло автоматическую модерацию",
    "details": {
      "reason": "Отсутствует фото для публичного места"
    }
  }
}
```

| **CSRF защита** | SameSite cookies для refresh token |
| **Rate Limiting** | 100 запросов/минута на пользователя |
| **Аудит** | Логирование всех действий CRUD с местами |
| **GDPR** | Право пользователя на удаление своих данных |

**Безопасность Яндекс Карт:**

## Бизнес-правила

### BR1: Типы доступа к местам

| Тип доступа | Видимость | Модерация | Рейтинг | Комментирование |
|------------|-----------|-----------|---------|-----------------|
| **Личное** | Только владелец | Не требуется | Нет | Нет |
| **Публичное** | Все авторизованные пользователи | Требуется | Да | Да |

**Правила:**
- Владелец может менять тип доступа в любое время
- При смене «Личное» → «Публичное»: отправка на модерацию
- При смене «Публичное» → «Личное»: скрытие с публичной карты

---

### BR2: Статусы мест

| Статус | Описание | Видимость |
|--------|----------|-----------|
| **active** | Место активно и отображается на карте | Публичное: всем; Личное: владельцу |
| **pending_moderation** | На модерации | Только владельцу и администраторам |
| **rejected** | Отклонено модератором | Только владельцу |
| **deleted** | Удалено | Никому (soft delete) |

**Правила переходов:**
- `pending_moderation` → `active` (одобрено)
- `pending_moderation` → `rejected` (отклонено)
- `active` → `pending_moderation` (редактирование публичного места)
- `active` → `deleted` (удаление)
- Любой статус → `deleted` (удаление владельцем или администратором)

---

### BR3: Модерация

**Правила:**
- Все публичные места проходят модерацию перед публикацией
- Администратор может одобрить или отклонить место
- При отклонении обязательно указать причину
- Владелец может повторно отправить место на модерацию после исправлений

**Критерии одобрения:**
- Название и описание не содержат спама и запрещенных слов
- Фотографии качественные и соответствуют месту
- Координаты находятся в допустимой зоне (не в океане)
- Минимум 1 тип рыбы выбран
- Минимум 1 фото загружено (для публичных)

**Сроки модерации:**
- SLA: 24 часа на рассмотрение
- Авто-отклонение за несоответствие критериям

---

### BR4: Типы рыб

**Правила:**
- Фиксированный список типов рыб (30+ видов)
- Минимум 1 тип при создании места
- Максимум 10 типов для одного места
- При поиске используется логика OR (карась ИЛИ лещ)

**Справочник типов рыб:**
- Хранится в отдельной таблице или как константы
- Каждый тип имеет id, имя, иконку, описание
- Поддержка мультиязычности (подготовка)

---

### BR5: Рейтинги

**Правила:**
- Рейтинг рассчитывается из оценок в отчетах (не из самой таблицы places)
- Шкала: 1-5 звезд
- Средний рейтинг: среднее арифметическое всех оценок
- Минимум 1 оценка для отображения рейтинга
- Рейтинг обновляется в реальном времени при добавлении новой оценки

**Формула:**
```
rating_avg = SUM(ratings) / COUNT(ratings)
```

**Триггер в БД:**
- Автоматическое обновление `rating_avg` и `reviews_count` при изменениях в таблице `ratings`

---

### BR6: Фото

**Правила:**
- Максимум 5 фото на место
- Форматы: JPEG, PNG, WEBP
- Размер каждого файла: до 10MB
- Автоматическая оптимизация через Cloudinary
- Первое фото — главное (превью)
- Обязательно минимум 1 фото для публичных мест

**Хранение:**
- Фото хранятся в Cloudinary
- URL сохраняются в массиве `images[]`
- Создаются thumbnails разных размеров

---

### BR7: Геолокация

**Правила:**
- Координаты: широта (latitude) -90° до 90°, долгота (longitude) -180° до 180°
- Точность: 6-8 знаков после запятой (10-15 метров)
- Валидация на backend и frontend
- Авто-определение геолокации при создании места (с разрешения пользователя)

---

### BR8: Уведомления

**Сценарии отправки уведомлений:**
1. Создание публичного места → Администраторам
2. Одобрение места → Владельцу
3. Отклонение места → Владельцу (с причиной)
4. Отчет привязан к месту → Владельцу места (опционально)

**Каналы:**
- Email (основной)
- Push-уведомления (будет реализовано позже)
- В-системные уведомления (будет реализовано позже)

---

### BR9: Квоты и ограничения

| Сущность | Ограничение | При превышении |
|----------|-------------|----------------|
| Мест на пользователя | 100 (личных) + 100 (публичных) | Платное расширение |
| Фото на место | 5 штук | Невозможно загрузить больше |
| Размер фото | 10MB | Ошибка загрузки |
| Типы рыб | 10 штук | Невозможно выбрать больше |
| Длина описания | 5000 символов | Валидация на frontend |

## Архитектура решения

### Обзор архитектуры

#### Архитектурные принципы

| Принцип | Описание | Реализация |
|---------|----------|------------|
| **Separation of Concerns** | Разделение ответственности между сервисами | Микросервисная архитектура (Auth, Places, Reports, Booking, Shop) |
| **Fail-Safe** | Отказоустойчивость при сбоях компонентов | Circuit Breaker, Retry Logic, Graceful Degradation |
| **Idempotency** | Возможность повторного выполнения операций | Уникальные идентификаторы, idempotency keys в API |
| **Eventual Consistency** | Согласованность данных во времени | Event-driven коммуникация между сервисами |
| **Security-First** | Безопасность как приоритет | JWT auth, HTTPS, Input validation, Rate limiting |
| **Scalability** | Масштабируемость по горизонтали | Stateless сервисы, горизонтальное масштабирование |
| **Observability** | Полная наблюдимость системы | Structured logging, Metrics, Distributed tracing |

### Идемпотентность и Resilience

#### Идемпотентные операции

```

1. Create Place:
   Header: X-Idempotency-Key: <UUID>
   Behavior: Returns existing place if key exists
   Retry-safe: Yes

2. Update Place:
   Header: X-Idempotency-Key: <UUID>
   Behavior: Replays if duplicate, no side effects
   Retry-safe: Yes

3. Rate Place:
   Header: X-Idempotency-Key: <UUID>
   Behavior: Updates or returns existing rating
   Retry-safe: Yes

   No idempotency key needed (idempotent by design)

   Retry-safe: Yes

### Event-Driven Architecture (Future Evolution)

#### Текущая архитектура vs Event-Driven

```
Текущая (Synchronous REST):

Frontend → Traefik → Places Service → PostgreSQL
                 ↓
              Reports Service → PostgreSQL
                 ↓
              Email Service → SMTP

Проблемы:
- Tight coupling between services
- Blocking operations
- Difficult to scale independently
- Single point of failure

Event-Driven (Future):

Frontend → Traefik → Places Service → PostgreSQL
                 ↓                ↓
            Event Bus (RabbitMQ/Kafka)
                 ↓                ↓
    ┌────────────┬────────────┬────────────┐
    │            │            │            │
Reports     Email        Analytics   Notification
Service     Service      Service      Service
    │            │            │            │
    └────────────┴────────────┴────────────┘

Преимущества:
- Loose coupling
- Asynchronous processing
- Better scalability
- Fault isolation
- Replay capability
```

#### События (Domain Events)

```
Domain Events:

1. PlaceCreated:
   - Trigger: New place created
   - Consumers: Reports (for stats), Email (for admin notification)
   
2. PlaceUpdated:
   - Trigger: Place modified
   - Consumers: Reports (update cache), Search (re-index)
   
3. PlaceModerated:
   - Trigger: Moderation action
   - Consumers: Email (notify owner), Search (publish/unpublish)
   
4. PlaceDeleted:
   - Trigger: Place deleted
   - Consumers: Reports (unlink), Analytics (archive)
   
5. ReportCreated:
   - Trigger: New report linked to place
   - Consumers: Places (update stats), Email (notify owner)
   
6. UserRegistered:
   - Trigger: New user
   - Consumers: Email (welcome email), Analytics (track signup)
```

6. Выбрать «Личное»
7. Нажать «Сохранить»

6. Нажать «Сохранить»
