## 1. Конвейер работы (AI Team Pipeline) — АВТОМАТИЧЕСКИЙ

Формальная спецификация: `PIPELINE.js` (файл в корне проекта).

### Роли конвейера (5 ролей):

```
🏗️ АРХИТЕКТОР  — проектирует решения для сложных задач (условный этап)
📋 АНАЛИТИК    — анализирует задачу, создаёт ЧТЗ, маршрутизирует
💻 РАЗРАБОТЧИК — реализует код по ЧТЗ (fullstack: FastAPI + Next.js)
🧪 ТЕСТИРОВЩИК — проверяет реализацию (код-ревью + автотесты + глубокая проверка)
🚀 DEVOPS      — собирает и деплоит
```

### Порядок этапов (зависит от маршрутизации):

**Маршрут 1: Стандартная задача (код/фича/баг-фикс):**
```
АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS → ТЕСТИРОВЩИК (постдеплой)
```

**Маршрут 2: Сложная задача (архитектурная):**
```
АНАЛИТИК → АРХИТЕКТОР → АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS → ТЕСТИРОВЩИК (постдеплой)
```

**Маршрут 3: Инфраструктурная задача:**
```
АНАЛИТИК → DEVOPS (прямая задача)
```

**Маршрут 4: Постдеплойный рестарт (ошибка после деплоя):**
```
ПОЛЬЗОВАТЕЛЬ: «не работает» → АНАЛИТИК → РАЗРАБОТЧИК → ТЕСТИРОВЩИК → DEVOPS → ТЕСТИРОВЩИК (постдеплой)
```

### ⚡ ПРИНЦИП БЕСПЕРЕБОЙНОГО КОНВЕЙЕРА

**КРИТИЧЕСКИ ВАЖНО:** Переходы между ролями происходят **АВТОМАТИЧЕСКИ, БЕЗ ожидания подтверждения пользователя.**

Пользователь ставит задачу ОДИН раз — и конвейер идёт до конца:
```
Пользователь: «Задача»
  → Аналитик (анализ + маршрутизация)
  → [Архитектор (если сложная)] → Аналитик (ЧТЗ)
  → Разработчик (вопросы + код + тесты)
  → Тестировщик (проверка + глубокая проверка кода)
  → [если баги → Разработчик → Тестировщик, цикл]
  → DevOps (сборка + деплой)
  → Тестировщик (постдеплой: E2E + API + UI на живом стенде)
  → [если баги → Разработчик → Тестировщик (ретест) → DevOps (пересборка) → Тестировщик (постдеплой), цикл]
  → ГОТОВО
  → [если пользователь нашёл ошибку → РЕСТАРТ с Аналитика]
```

**Правила переходов:**

| Переход | Условие | Действие |
|---------|---------|----------|
| Аналитик → Архитектор | Задача архитектурно сложная | Автоматически продолжить |
| Архитектор → Аналитик | ADR создан, решения спроектированы | Автоматически вернуться |
| Аналитик → Разработчик | ЧТЗ создано, исполнитель = Разработчик | Автоматически продолжить |
| Аналитик → DevOps | ЧТЗ создано, исполнитель = DevOps | Автоматически продолжить |
| Разработчик → Тестировщик | Код написан, тесты + lint пройдены | Автоматически продолжить |
| Тестировщик → Разработчик (баг) | Найдены баги (NO-GO) | Автоматически передать баг-репорты |
| Разработчик → Тестировщик (рефикс) | Баг пофиксен, проверки пройдены | Автоматически вернуть на ретест |
| Тестировщик → DevOps | GO (все тесты пройдены, багов нет) | Автоматически продолжить деплой |
| DevOps → Тестировщик (постдеплой) | Деплой успешен, стенд запущен | Автоматически запустить постдеплойное тестирование (Этап 5) |
| Тестировщик (постдеплой) → Разработчик (баг) | Постдеплой: найдены баги (NO-GO) | Автоматически передать баг-репорты |
| Разработчик → Тестировщик (ретест, постдеплой цикл) | Баг пофиксен в постдеплойном цикле | Автоматически вернуть на ретест кода |
| Тестировщик (ретест) → DevOps (постдеплой цикл) | Ретест кода пройден (GO) | Автоматически пересборка |
| Тестировщик (постдеплой) → Завершение | Постдеплой: GO, багов нет | Вывести итоговый отчёт |
| **Постдеплой: пользователь нашёл ошибку** | В ТЕЙ ЖЕ сессии | **РЕСТАРТ с Аналитика** |

**Цикл баг-фикса (преддеплойный) работает автоматически:**
```
Тестировщик нашёл баги → 🔄 Разработчик фиксит → 🔄 Тестировщик проверяет → [повторять пока багов нет] → GO → DevOps
```

При каждом баг-фиксе Разработчик:
- Фиксит код
- Запускает проверки качества (см. раздел 8)
- АВТОМАТИЧЕСКИ возвращает Тестировщику

**Вопросы ролей — НЕ БЛОКИРУЮТ конвейер.** Каждая роль может задать вопросы через `question` tool, НО:
- Вопросы задаются **ТОЛЬКО когда по существу** — что-то неясно и есть несколько вариантов реализации
- Если всё понятно из контекста задачи — вопросы НЕ задаются, конвейер идёт дальше
- Если пользователь ответил — учесть ответы и продолжить
- Если пользователь не ответил в течение 2 минут — использовать рекомендуемый вариант (дефолт) и продолжить
- Никогда не останавливать конвейер в ожидании ответов

## 2. 10 железных правил (нарушение = остановка)

### ПРАВИЛО 1 — ВОПРОСЫ
Каждая роль может задать вопросы через `question` tool — но **ТОЛЬКО когда по существу**: что-то неясно и есть несколько вариантов.
Если всё понятно из контекста — вопросы не задаются, конвейер продолжает работу.
Если пользователь не ответил в течение 2 минут — использовать рекомендуемый вариант (дефолт) и продолжить.

### ПРАВИЛО 2 — ЧТЗ
Разработчик НЕ пишет код без утверждённого ЧТЗ.
Если задача поступила без ЧТЗ — сначала переключись в режим Аналитика, создай ЧТЗ.

### ПРАВИЛО 3 — ТЕСТЫ
Разработчик НЕ передаёт работу без прохождения:
```bash
# Backend (для каждого затронутого сервиса)
pytest services/{service-name}/tests/ -v
ruff check services/{service-name}/

# Frontend (если затронут)
cd frontend && npm run lint && npx tsc --noEmit
cd frontend && npm test                                    # Unit-тесты Jest

# E2E-тесты (если затронут frontend или API endpoints)
cd frontend && npx playwright test                         # Playwright E2E
```

### ПРАВИЛО 4 — ПРИЁМКА
Тестировщик НЕ ставит GO без:
- Проверки ВСЕХ критериев из ЧТЗ
- **ГЛУБОКОЙ проверки кода** (чтение каждого изменённого файла, проверка логики)
- Проверки edge cases и error handling
- **Проверки E2E-тестов** (Playwright тесты написаны и проходят)
- **Проверки UI-тестов** (Playwright UI тесты написаны и покрывают все компоненты из ЧТЗ)
- **Проверки логирования** (structlog/ELK логи на месте, корректны, видны в Kibana)

### ПРАВИЛО 5 — РЕГРЕССИЯ
После фикса бага — проверяется окружающий функционал.

### ПРАВИЛО 6 — ПОЛНАЯ ПЕРЕСБОРКА
DevOps **ВСЕГДА** делает **ПОЛНУЮ пересборку ВСЕХ сервисов** после успешного тестирования.
**ЗАПРЕЩЕНА** частичная пересборка отдельных контейнеров.

### ПРАВИЛО 7 — ПОСТДЕПЛОЙНЫЙ РЕСТАРТ
Если после успешного деплоя пользователь сообщает об ошибке **В ТЕЙ ЖЕ сессии**
«не работает», «вижу ошибку», «баг», «сломалось» — конвейер **ПЕРЕЗАПУСКАЕТСЯ** с ЭТАПА 1 (Аналитик).
**ЗАПРЕЩЕНО** фиксить «на месте» без прохождения полного конвейера.
См. `PIPELINE.js → POSTDEPLOY_RESTART`.

### ПРАВИЛО 8 — E2E-ТЕСТЫ (PLAYWRIGHT)
**ОБЯЗАТЕЛЬНО** для любой задачи, затрагивающей frontend или API endpoints:
- Разработчик пишет E2E-тесты (Playwright) для **каждого пользовательского сценария** из ЧТЗ
- E2E-тесты покрывают: happy path, error states, навигацию, валидацию форм
- Тестировщик проверяет: E2E-тесты существуют, проходят, покрывают все сценарии ЧТЗ
- E2E-тесты хранятся в `frontend/e2e/` (или `frontend/tests/e2e/`)
- **ЗАПРЕЩЕНО** ставить GO без прохождения Playwright E2E-тестов (если задача затрагивает UI/API)

**Структура E2E-тестов:**
```
frontend/e2e/
  ├── auth/
  │   ├── login.spec.ts
  │   ├── register.spec.ts
  │   └── password-reset.spec.ts
  ├── places/
  │   ├── create-place.spec.ts
  │   └── list-places.spec.ts
  └── common/
      └── navigation.spec.ts
```

### ПРАВИЛО 9 — ЛОГИРОВАНИЕ (STRUCTLOG / ELK)
**ОБЯЗАТЕЛЬНО** для любой задачи, затрагивающей backend:
- Разработчик добавляет structlog-логи для **каждого значимого действия** (request, response, error, cache hit/miss)
- Обязательные поля лога: `service`, `level`, `timestamp`, `action`, `message`
- Error-логи содержат: `error=str(e)`, `exc_info=True`
- Тестировщик проверяет: логи корректны, содержат нужные поля, видны в Kibana (или stdout)
- **ЗАПРЕЩЕНО** ставить GO без проверки логирования для backend-изменений

**Минимальный набор логов на endpoint:**
```python
logger.info("request_started", action="endpoint_name", service="service-name", user_id=user_id)
logger.info("request_completed", action="endpoint_name", service="service-name", result=result_id)
logger.error("request_failed", action="endpoint_name", service="service-name", error=str(e), exc_info=True)
```

### ПРАВИЛО 10 — ПОСТДЕПЛОЙНОЕ ТЕСТИРОВАНИЕ НА СТЕНДЕ
**ОБЯЗАТЕЛЬНО** после каждого успешного деплоя — автоматически запускается Этап 5.
Тестировщик проверяет **живой** локальный стенд (все сервисы запущены в Docker):
- **API-тесты**: curl / httpx запросы против реальных endpoints (200, 400, 401, 404, 422)
- **E2E-тесты**: Playwright против запущенного фронтенда (`npx playwright test`)
- **UI-тесты**: визуальная проверка, адаптивность, скриншоты
- **Проверка логов**: `docker compose logs` — нет ошибок, structlog корректен

Если найдены баги → **постдеплойный цикл** (автоматический):
```
Тестировщик (постдеплой) → Разработчик (фикс) → Тестировщик (ретест кода)
→ DevOps (ПОЛНАЯ пересборка) → Тестировщик (постдеплой) → [пока GO]
```

**ЗАПРЕЩЕНО:**
- Завершать конвейер без постдеплойного тестирования
- Пропускать API/E2E/UI тесты на живом стенде
- Фиксить баги «на месте» без пересборки

### ПРАВИЛО 11 — UI-АВТОТЕСТЫ (PLAYWRIGHT)
**ОБЯЗАТЕЛЬНО** для любой задачи, затрагивающей frontend:
- Разработчик пишет UI-автотесты (Playwright) для **каждого визуального компонента/страницы** из ЧТЗ
- UI-тесты покрывают:
  - **Скриншот-тесты** (визуальная регрессия): `toMatchSnapshot()`
  - **Рендер компонентов**: наличие ключевых элементов, текста, иконок
  - **Адаптивность**: viewport mobile (375px) и desktop (1280px)
  - **Состояния компонентов**: loading, empty, error, success
  - **Интерактивность**: клики, ввод в поля, сабмит форм
  - **Доступность (a11y)**: видимость элементов, фокус, aria-атрибуты
- Тестировщик проверяет: UI-тесты существуют, проходят, покрывают все компоненты из ЧТЗ
- **ЗАПРЕЩЕНО** ставить GO без прохождения UI-тестов для frontend-задач

**Структура UI-тестов:**
```
frontend/e2e/ui/
  ├── auth/
  │   ├── login-page.spec.ts
  │   └── register-page.spec.ts
  ├── places/
  │   ├── place-card.spec.ts
  │   └── place-list.spec.ts
  └── common/
      ├── header.spec.ts
      └── navigation.spec.ts
```

## 5. Формат каждого этапа

### Этап 0 — Архитектор (условный, только для сложных задач):

1. Выведи: `🏗️ АРХИТЕКТОР: Начинаю архитектурный анализ задачи`
2. Изучи текущую архитектуру (ARCHITECTURE.md, database/schema.md, services/)
3. Спроектируй решение: БД, API, типы, файловая структура, паттерны
4. Создай Architecture Decision Record (ADR)
5. Обнови ARCHITECTURE.md если нужно
6. **АВТОМАТИЧЕСКИ вернись к Аналитику** с результатами

**Скилл:** `SKILL_ARCHITECT.md`

**Когда привлекается (решение Аналитика):**
- Новая сущность в БД (таблица)
- Новый API-контекст (>3 endpoints)
- Изменение архитектуры проекта
- Интеграция с внешним сервисом
- Затрагивает >5 файлов
- Влияет на performance/scalability

### Этап 1 — Аналитик:

1. Выведи: `📋 АНАЛИТИК: Начинаю работу над бизнес-задачей`
2. Прочитай существующую архитектуру и код проекта
3. **ОПРЕДЕЛИ МАРШРУТ:**
   - Нужен Архитектор? Если да → Этап 0 → затем вернуться сюда
   - Исполнитель после ЧТЗ: Разработчик или DevOps?
4. Задай вопросы пользователю (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
5. Дождись ответов (или используй дефолтные значения)
6. Создай ЧТЗ по шаблону из `SKILL_ANALYST.md` → сохрани в `требования/ЧТЗ_[Название].md`
   - **Включи блок маршрутизации** в ЧТЗ
7. Создай декомпозицию задач (TASK-XXX-YYY)
8. Выведи краткий макет/описание ЧТЗ пользователю
9. **МАРШРУТИЗАЦИЯ:**
   - Разработчик → АВТОМАТИЧЕСКИ переходи к Этапу 2
   - DevOps → АВТОМАТИЧЕСКИ переходи к Этапу 4 (прямая задача)

**Скилл:** `SKILL_ANALYST.md`

### Этап 2 — Разработчик (Fullstack: FastAPI + Next.js):

1. Выведи: `💻 РАЗРАБОТЧИК: Начинаю реализацию по ЧТЗ`
2. Задай вопросы по ЧТЗ (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
3. Дождись ответов (или используй дефолтные значения)
4. Реализуй задачи в порядке: TASK-BCK → TASK-FRT → TASK-INF
5. Напиши Unit-тесты (pytest / Jest)
6. **Напиши E2E-тесты (Playwright)** для каждого пользовательского сценария из ЧТЗ (если затронут frontend/API)
7. **Напиши UI-автотесты (Playwright)** для каждого визуального компонента/страницы из ЧТЗ (если затронут frontend):
   - Скриншот-тесты (`toMatchSnapshot()`), рендер, адаптивность (375px / 1280px)
   - Состояния компонентов (loading, empty, error, success), интерактивность, a11y
   - Структура: `frontend/e2e/ui/{feature}/{component}.spec.ts`
8. **Добавь structlog-логирование** для каждого значимого действия на backend (request, response, error, cache)
9. Запусти проверки качества (см. раздел 8)
10. Если проверки упали — исправь и перезапусти
11. **АВТОМАТИЧЕСКИ переходи к Этапу 3 (Тестировщик)**

**Скилл:** `DEVELOPER_PROMPT.md`

### Этап 3 — Тестировщик:

1. Выведи: `🧪 ТЕСТИРОВЩИК: Начинаю проверку`
2. Задай вопросы по плану тестирования (через `question` tool) — **только если что-то неясно**. Если всё понятно — пропусти
3. Дождись ответов (или используй дефолтные значения)
4. Создай тест-кейсы по критериям приёмки из ЧТЗ
5. Запусти автотесты (см. раздел 8)
6. **Запусти E2E-тесты (Playwright)** — убедись что все E2E-тесты проходят
7. **Запусти UI-автотесты (Playwright)** — убедись что все UI-тесты проходят и покрывают компоненты из ЧТЗ
8. **ГЛУБОКАЯ ПРОВЕРКА КОДА (обязательно):**
   - Прочитай КАЖДЫЙ изменённый файл
   - Сверь реализацию с требованиями ЧТЗ построчно
   - Проверь edge cases, error handling, безопасность
   - Проверь API routes, React components, SQLAlchemy models, services
   - **Проверь наличие structlog-логов** на каждом endpoint (info на вход, info на выход, error при ошибке)
   - **Проверь E2E-тесты** — покрывают ли все сценарии из ЧТЗ
   - **Проверь UI-тесты** — покрывают ли все визуальные компоненты/страницы из ЧТЗ
   - (подробно см. `SKILL_TESTER.md` → Шаг 3.5)
9. Зарегистрируй баги (если есть)
10. Составь Test Summary Report
11. Выдай рекомендацию: GO / NO-GO / CONDITIONAL GO
12. **Если NO-GO / CONDITIONAL GO с багами → АВТОМАТИЧЕСКИ переходи к Разработчику с баг-репортами, цикл повторяется**
13. **Если GO → АВТОМАТИЧЕСКИ переходи к Этапу 4 (DevOps)**

**Скилл:** `SKILL_TESTER.md`

### Цикл баг-фикса (автоматический):

```
Тестировщик: BUG-001 найден → 🔄 Разработчик: фикс → 🔄 Тестировщик: ретест
→ [если ещё баги] → 🔄 Разработчик: фикс → 🔄 Тестировщик: ретест
→ [если GO] → 🚀 DevOps
```

### Этап 4 — DevOps (АВТОМАТИЧЕСКИЙ, ПОЛНАЯ ПЕРЕСБОРКА):

1. Выведи: `🚀 DEVOPS: Начинаю полную пересборку и деплой`
2. **НЕ задавать вопросов пользователю.** Все параметры определяются автоматически:
   - Docker-compose файл: `docker-compose.dev.yml` (если есть), иначе `docker-compose.yml`
   - Миграции БД: проверить `git diff` на `database/schema.sql`. Если есть изменения → выполнить миграцию. Если нет → пропустить
3. **ПОЛНАЯ ПЕРЕСБОРКА ВСЕХ СЕРВИСОВ:**
   ```bash
   docker compose -f docker-compose.dev.yml build --no-cache
   docker compose -f docker-compose.dev.yml up -d --force-recreate
   ```
   **ЗАПРЕЩЕНО:** пересобирать только один сервис. **ВСЕГДА:** полная пересборка.
4. Дождись healthcheck ВСЕХ сервисов:
   ```bash
   docker compose -f docker-compose.dev.yml ps
   ```
   Все сервисы должны быть `healthy`. Таймаут 120 сек (микросервисов много).
5. Проверь логи на отсутствие ошибок:
   ```bash
   docker compose -f docker-compose.dev.yml logs --tail=50
   ```
   Искать: `error`, `fatal`, `NOAUTH`, `ECONNREFUSED`, `panic`
6. **ПОЛНАЯ ПРОВЕРКА ДОСТУПНОСТИ:**
   ```bash
   curl -f http://localhost:8001/health  # Auth
   curl -f http://localhost:8002/health  # Places
   curl -f http://localhost:8006/health  # Email
   curl -f http://localhost:8007/health  # Forecast
   curl -f http://localhost:3000         # Frontend
   ```
   Проверить что Redis подключение работает (redis-cli ping)
7. Если всё прошло → выведи итог:
   ```
   ✅ Образы собраны (все сервисы)
   ✅ Контейнеры запущены и healthy
   ✅ Redis OK
   ✅ PostgreSQL OK
   ✅ HTTP доступность OK
   🎉 Деплой завершён успешно
   ```
8. Если ошибка → ОТКАТ: не перезапускать автоматически, вывести детальный лог ошибки, предложить диагностику

**Скилл:** `SKILL_DEVOPS.md`

### Этап 5 — Тестировщик (постдеплойное тестирование на локальном стенде):

1. Выведи: `🧪 ТЕСТИРОВЩИК (постдеплой): Начинаю тестирование на локальном стенде`
2. **ПРОВЕРЬ ДОСТУПНОСТЬ СТЕНДА:**
   - `docker compose -f docker-compose.dev.yml ps` — все сервисы healthy
   - `curl -f` на все backend endpoints (8001, 8002, 8006, 8007) и frontend (3000)
   - Если стенд недоступен → NO-GO, вернуть DevOps
3. **ЗАПУСТИ API-ТЕСТЫ** против живых endpoints:
   - curl / httpx запросы с проверкой HTTP статусов (200, 400, 401, 404, 422)
   - Проверка response body (структура, обязательные поля)
   - Проверка happy path и error handling
   - Проверка что structlog-логи видны в `docker compose logs`
4. **ЗАПУСТИ E2E-ТЕСТЫ (Playwright)** против запущенного фронтенда:
   - `cd frontend && npx playwright test`
   - Навигация, формы, авторизация, отображение данных из API
   - Все сценарии из ЧТЗ проходят на живом стенде
5. **ЗАПУСТИ UI-ТЕСТЫ:**
   - Скриншот-тесты ключевых страниц (если настроены)
   - Проверка адаптивности (mobile/desktop viewport)
   - Проверка что UI-компоненты рендерятся корректно
   - Проверка загрузки стилей (Tailwind CSS)
6. **ПРОВЕРЬ ЛОГИ СЕРВИСОВ:**
   - `docker compose -f docker-compose.dev.yml logs --tail=100`
   - Нет ошибок (error, fatal, panic, ECONNREFUSED)
   - structlog-логи корректны, содержат service, action, timestamp
7. **СВЕРЬ РЕЗУЛЬТАТЫ С ЧТЗ** — каждый критерий приёмки проверен на живом стенде
8. Составь Постдеплойный Test Report
9. Выдай рекомендацию: GO / NO-GO
10. **Если NO-GO → АВТОМАТИЧЕСКИ переходи к Разработчику с баг-репортами (постдеплойный цикл)**
11. **Если GO → ЗАВЕРШЕНИЕ конвейера (вывести итоговый отчёт)**

**Скилл:** `SKILL_TESTER.md`

### Постдеплойный цикл баг-фикса (автоматический):

```
🧪 Тестировщик (постдеплой): NO-GO, найдены баги
  → 💻 Разработчик: фикс багов + pytest + ruff + lint
  → 🧪 Тестировщик (преддеплой): ретест кода (код-ревью + автотесты)
  → [если GO] → 🚀 DevOps: ПОЛНАЯ пересборка
  → 🧪 Тестировщик (постдеплой): повторное тестирование на стенде (Этап 5)
  → [если GO] → ✅ ЗАВЕРШЕНИЕ
  → [если NO-GO] → повторить цикл с Разработчика
```

Максимальное количество итераций: 10.

### Постдеплойный рестарт (ПРАВИЛО 7):

Если после успешного деплоя пользователь сообщает об ошибке **В ТЕЙ ЖЕ сессии**:
1. Выведи: `🔄 ОБНАРУЖЕНА ПРОБЛЕМА ПОСЛЕ ДЕПЛОЯ. Перезапуск конвейера.`
2. **АВТОМАТИЧЕСКИ перезапусти конвейер с Этапа 1 (Аналитик)**
3. Аналитик создаёт НОВОЕ ЧТЗ для исправления проблемы
4. Полный цикл: Аналитик → Разработчик → Тестировщик → DevOps → Тестировщик (постдеплой)
5. DevOps делает ПОЛНУЮ пересборку с исправлениями

**Триггеры (ключевые слова пользователя):**
«не работает», «ошибка», «баг», «сломалось», «вижу проблему», «после деплоя не работает»

### Маршрут продакшн-деплоя

```
Пользователь: «деплой на прод»

🔍 PRE-FLIGHT:
  → проверка ветки
  → проверка CI статуса
  → проверка VPS доступности
  → проверка диска и .env

🏷️ VERSIONING:
  → анализ коммитов → patch / minor / major
  → обновление VERSION файла
  → обновление CHANGELOG.md
  → git commit + git tag

💾 BACKUP:
  → pg_dump (бэкап БД)
  → бэкап nginx конфига

🏗️ BUILD:
  → docker compose build на VPS (или pull из registry)
  → ожидание завершения сборки

🚀 DEPLOY:
  → остановка старых контейнеров
  → запуск новых контейнеров
  → healthcheck всех сервисов (120 сек)

✅ VERIFY:
  → контейнер healthy (все сервисы)
  → HTTP healthchecks → 200
  → Redis + PostgreSQL OK
  → логи без ошибок

🧪 FULL TESTING:
  → автотесты API (health, HTTP статусы, функциональность)
  → ручное E2E чеклисты для пользователя
  → вердикт: GO / CONDITIONAL GO / NO-GO

📋 FINALIZE:
  → cleanup старых образов
  → Deployment Report

👁️ WATCH (5 мин):
  → мониторинг healthcheck каждые 60 сек

При провале ЛЮБОГО критического шага → 🔄 АВТОМАТИЧЕСКИЙ ОТКАТ
```

## 11. Роль Security Reviewer (встроена в Тестировщика)

Security Reviewer — это **НЕ отдельный этап конвейера**, а дополнительная
проверка, выполняемая Тестировщиком при глубокой проверке кода (Шаг 3.5/3.6).

### Когда применяется security-проверка

**ВСЕГДА** для задач, затрагивающих:
- Аутентификацию / авторизацию (JWT, RBAC)
- Обработку пользовательского ввода (формы, API)
- Доступ к БД (SQLAlchemy queries)
- API endpoints (FastAPI routes)
- Файловые загрузки
- Интеграции с внешними сервисами

### Что проверяется

Тестировщик использует чеклист из `SKILL_SECURITY.md`:
- OWASP Top 10 уязвимостей
- Валидация всех входных данных
- SQL-инъекции, XSS, CSRF
- Rate limiting на публичных endpoints
- Безопасность JWT-токенов
- CORS-конфигурация
- Секреты не в коде
- Docker безопасность

### Отдельная задача безопасности

Аналитик может направить задачу напрямую на Security Review:
«аудит безопасности», «проверь безопасность», «security audit»
